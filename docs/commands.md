# Команды git

## Сокращения и обозначения

```bash
[] - опциональный аргумент
<name> - имя сущности (ветка, файл, путь и т.д.)
```

## Пошаговая работа с репозиторием

### Шаг 1. Инициализация репозитория

**Когда использовать:** начинаешь новый проект локально (без удаленного репозитория).

```bash
mkdir <repo>
```
Создает папку проекта.

```bash
cd <repo>
```
Переходит в рабочую директорию проекта.

```bash
git init
```
Инициализирует Git-репозиторий в текущей папке и создает скрытую директорию `.git/`.

```bash
git status
```
Показывает текущее состояние репозитория: ветку, отслеживаемые и неотслеживаемые файлы.

**Результат шага:** создан локальный репозиторий, готовый к первому коммиту.

### Шаг 2. Добавление файлов в индекс и создание коммита

**Когда использовать:** после изменений в файлах, перед фиксацией в истории.

```bash
git status
```
Показывает, какие файлы изменены и какие уже в индексе (staged).

```bash
git add <file>
```
Добавляет конкретный файл в индекс. Для всех изменений можно использовать `git add .`.

```bash
git ls-files --stage
```
Показывает содержимое индекса: какие файлы подготовлены и с какими blob-хэшами.

```bash
git diff
```
Сравнивает рабочую директорию с индексом (что изменено, но **еще не добавлено** через `git add`).
Если команда ничего не выводит — все текущие правки уже попали в индекс.

```bash
git diff --cached
```
Сравнивает индекс с последним коммитом (`HEAD`) и показывает, **что именно попадет в новый коммит**.
Это ключевая проверка перед `git commit`, особенно если изменений много.

```bash
git commit -m "<message>"
```
Создает коммит из текущего состояния индекса.

**Результат шага:** нужные изменения добавлены в индекс и зафиксированы в истории.

### Шаг 3. Работа с удаленным репозиторием

**Когда использовать:** при публикации локальных коммитов, синхронизации и работе с внешними репозиториями.

#### 3.1 Проверка и добавление удаленных репозиториев

```bash
git remote -v
```
Показывает список подключенных удаленных репозиториев и их URL.

```bash
git remote add origin https://github.com/<user>/<repo>.git
```
Добавляет основной удаленный репозиторий `origin` по HTTPS.

```bash
git remote add origin git@github.com:<user>/<repo>.git
```
Добавляет `origin` по SSH.

```bash
git remote add upstream https://github.com/<owner>/<repo>.git
```
Добавляет второй удаленный репозиторий `upstream` (часто используется при работе с форком).

```bash
git remote set-url origin git@github.com:<user>/<repo>.git
```
Меняет URL существующего `origin`, например с HTTPS на SSH.

#### 3.2 Назначение origin и upstream

- `origin` — основной удаленный репозиторий локального проекта (обычно собственный репозиторий или форк).
- `upstream` — исходный репозиторий, откуда регулярно подтягиваются изменения.
- Имена `origin` и `upstream` — соглашение, а не обязательное правило; технически можно использовать любое имя.

#### 3.3 Удаленные ветки и `origin/main`

```bash
git fetch origin
```
Обновляет сведения о ветках на удаленном `origin` без изменения рабочей директории.

```bash
git branch -r
```
Показывает удаленные ветки (`origin/main`, `origin/dev`, `upstream/main` и т.д.).

`origin/main` — это **remote-tracking branch** в локальном репозитории:
- хранит локальный снимок состояния ветки `main` на сервере `origin`;
- обновляется командами `git fetch` и `git pull`;
- используется как опорная точка для сравнения и синхронизации (`git diff main origin/main`).

#### 3.4 Публикация и синхронизация

```bash
git push -u origin main
```
Публикует локальную ветку `main` в `origin` и связывает ее с удаленной веткой.

```bash
git pull --rebase origin main
```
Подтягивает изменения из `origin/main` и применяет локальные коммиты поверх обновленной истории.

```bash
git fetch upstream
git merge upstream/main
```
Обновляет локальную ветку изменениями из исходного репозитория `upstream`.

**Результат шага:** локальный репозиторий связан с удаленными источниками, настроены `origin`/`upstream`, выполнена базовая синхронизация веток.

### Шаг 4. Работа с ветками

**Когда использовать:** при параллельной разработке задач, изоляции изменений и подготовке к слиянию.

#### 4.1 Просмотр и создание веток

```bash
git branch
```
Показывает локальные ветки и отмечает текущую ветку.

```bash
git branch <new-branch>
```
Создает новую ветку от текущего коммита.

```bash
git branch <new-branch> <commit>
```
Создает ветку от указанного коммита (удобно для возврата к нужной точке истории).

#### 4.2 Переключение между ветками

```bash
git checkout <branch>
```
Переключает рабочую директорию на выбранную ветку.

```bash
git checkout -b <new-branch>
```
Создает новую ветку и сразу переключает на нее.

```bash
git checkout -m <branch>
```
Пытается переключить ветку с сохранением локальных незакоммиченных изменений.

#### 4.3 Слияние веток

```bash
git checkout main
git merge <feature-branch>
```
Вливает изменения из рабочей ветки в `main`.

```bash
git merge-base main <feature-branch>
```
Показывает общий базовый коммит двух веток. Полезно при анализе истории и сложных конфликтов.

```bash
git diff main...<feature-branch>
```
Показывает различия относительно общей базы веток (удобно для проверки объема изменений перед merge).

#### 4.4 Удаление и очистка веток

```bash
git branch -d <branch>
```
Удаляет локальную ветку, если она уже слита.

```bash
git branch -D <branch>
```
Принудительно удаляет локальную ветку, даже если она не слита.

```bash
git push origin --delete <branch>
```
Удаляет удаленную ветку в `origin`.

**Результат шага:** ветки используются для изоляции задач, изменения безопасно переключаются и корректно сливаются в основную линию разработки.

### Шаг 5. Merge-конфликты: обнаружение и разрешение

**Когда использовать:** при `merge`, `pull` или `rebase`, если Git не может автоматически объединить изменения.

#### 5.1 Как понять, что есть конфликт

```bash
git status
```
Показывает файлы в конфликте (`both modified` и похожие статусы).

```bash
git diff --name-only --diff-filter=U
```
Выводит только список неразрешенных конфликтов.

Внутри конфликтного файла появляются маркеры:
- `<<<<<<< HEAD` — текущая версия;
- `=======` — разделитель;
- `>>>>>>> <branch>` — версия из вливаемой ветки.

#### 5.2 Базовый порядок разрешения

```bash
git merge <feature-branch>
```
Запускает слияние и может остановиться на конфликте.

```bash
git checkout --ours <file>
```
Оставляет локальную версию файла.

```bash
git checkout --theirs <file>
```
Оставляет версию из вливаемой ветки.

```bash
git add <file>
```
Помечает конфликт как решенный для конкретного файла.

```bash
git merge --continue
```
Продолжает merge после разрешения всех конфликтов.

```bash
git merge --abort
```
Отменяет merge и возвращает состояние до начала слияния.

Для `rebase` используются команды `git rebase --continue` и `git rebase --abort`.

#### 5.3 Подключение инструментов разрешения конфликтов

##### Meld

```bash
git config --global merge.tool meld
git config --global mergetool.prompt false
git config --global mergetool.keepBackup false
git mergetool
```
Настраивает `meld` как инструмент merge и запускает визуальное разрешение конфликтов.

##### Vimdiff

```bash
git config --global merge.tool vimdiff
git config --global mergetool.vimdiff.layout "LOCAL,BASE,REMOTE / MERGED"
git mergetool
```
Настраивает `vimdiff` для четырехпанельного сравнения и редактирования.

#### 5.4 Окружение и временное переключение инструмента

```bash
GIT_MERGETOOL=vimdiff git mergetool
```
Временно запускает `mergetool` через `vimdiff`, без изменения глобальной конфигурации.

```bash
export VISUAL=vim
export EDITOR=vim
```
Задает редактор для сообщений merge/rebase и ручного редактирования конфликтных файлов.

**Результат шага:** конфликты обнаруживаются заранее, разрешаются по понятной схеме и при необходимости обрабатываются через визуальные инструменты.

### Шаг 6. Конфигурация Git и правила игнорирования

**Когда использовать:** при первичной настройке Git, стандартизации поведения и фильтрации служебных файлов.

#### 6.1 Три уровня конфигурации Git

Git читает конфигурацию каскадно (от общего к частному):
1. **System** — системный уровень, обычно `/etc/gitconfig`.
2. **Global** — пользовательский уровень, обычно `~/.gitconfig`.
3. **Local** — уровень конкретного репозитория, файл `.git/config`.

Локальные настройки имеют наивысший приоритет и переопределяют глобальные и системные.

```bash
git config --list --show-origin
```
Показывает все активные настройки и путь к файлу, откуда взято каждое значение.

```bash
git config --system user.name "Team Default"
```
Записывает значение на системном уровне (для всех пользователей машины).

```bash
git config --global user.name "Alice Dev"
git config --global user.email "alice@example.com"
```
Записывает имя и email на уровне текущего пользователя.

```bash
git config --local user.name "Project Bot"
```
Записывает значение только для текущего репозитория (`.git/config`).

#### 6.2 Где размещать `.gitignore`

- `./.gitignore` в корне репозитория — базовые правила проекта.
- `./subdir/.gitignore` во вложенных папках — локальные правила для конкретной директории и ее поддерева.
- `.git/info/exclude` — локальные исключения только для текущего репозитория (не коммитится).
- Глобальный ignore-файл пользователя (через `core.excludesFile`) — общие личные правила для всех репозиториев, например `~/.gitignore_global`.

```bash
git config --global core.excludesFile ~/.gitignore_global
```
Подключает глобальный файл игнорирования.

#### 6.3 Каскадные правила и отмена через `!`

Каскадные правила означают, что Git проверяет несколько источников правил; более специфичные и более поздние совпадения переопределяют предыдущие.

Пример:

```gitignore
# Игнорировать все логи
*.log

# Но сохранить важный лог в репозитории
!important.log
```

`!pattern` отменяет предыдущее игнорирование для совпадающего пути.

**Результат шага:** конфигурация Git разделена по уровням, правила игнорирования управляются предсказуемо и без лишнего шума в `git status`.

### Шаг 7. Откат изменений

**Когда использовать:** при ошибочных правках, неверном добавлении в индекс или необходимости вернуть файл к предыдущей версии.

#### 7.1 Откат файлов, добавленных в индекс (unstage)

```bash
git restore --staged <file>
```
Убирает файл из индекса, но оставляет изменения в рабочей директории.

```bash
git reset HEAD <file>
```
Альтернативный способ снять файл с индекса (часто встречается в старых инструкциях).

#### 7.2 Откат файлов, не добавленных в индекс

```bash
git restore <file>
```
Отменяет незакоммиченные изменения файла в рабочей директории и возвращает состояние из индекса.

```bash
git restore .
```
Отменяет все незакоммиченные изменения в текущей директории и вложенных папках.

#### 7.3 Полный откат файла: и из индекса, и из рабочей директории

```bash
git restore --staged <file>
git restore <file>
```
Последовательно убирает файл из индекса и затем откатывает его содержимое в рабочей директории.

#### 7.4 Откат конкретного файла к состоянию из конкретного коммита

```bash
git restore --source <commit> -- <file>
```
Берет версию файла из указанного коммита и записывает ее в рабочую директорию.

```bash
git add <file>
git commit -m "restore <file> from <commit>"
```
Фиксирует восстановленную версию отдельным коммитом.

Пример:

```bash
git restore --source a1b2c3d -- docs/commands.md
```
Восстанавливает `docs/commands.md` из коммита `a1b2c3d`.

**Результат шага:** изменения откатываются точно на нужном уровне — индекс, рабочая директория или конкретная версия файла из истории.
